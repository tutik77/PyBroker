# ПРАКТИКА: РАЗРАБОТКА БРОКЕРА СООБЩЕНИЙ С НУЛЯ

## ЦЕЛЬ ПРОЕКТА
Разработать собственный брокер сообщений с базовой функциональностью pub/sub, очередями сообщений.

## ОПИСАНИЕ ЗАДАЧИ
Необходимо реализовать брокер сообщений, который будет обеспечивать асинхронную коммуникацию между тремя тестовыми микросервисами (один publisher, два subsriber).

"Дополнительно" - реализовать клиентскую библиотеку для работы с вашим брокером (на любом ЯП).

## ОСНОВНЫЕ ТРЕБОВАНИЯ

### 1. БАЗОВАЯ ФУНКЦИОНАЛЬНОСТЬ (Обязательно)
- Реализация паттерна Publisher/Subscriber
- Поддержка очередей сообщений (FIFO, LIFO ...)
- Возможность публикации сообщений в топики/очереди
- Возможность подписки на топики/очереди
- Обработка множественных подписчиков на один топик/очередь

### 2. ПЕРСИСТЕНТНОСТЬ (Обязательно)
- Сохранение сообщений на диск (файловая система или БД)
- Восстановление состояния после перезапуска
- Гарантия доставки сообщений (at-least-once или exactly-once)

### 3. КЛИЕНТСКАЯ БИБЛИОТЕКА (Обязательно)
- SDK для любого ЯП

### 4. ДОПОЛНИТЕЛЬНЫЕ ФУНКЦИИ (просто мои идеи)
- Приоритеты сообщений
- TTL (Time To Live) для сообщений
- Dead Letter Queue (DLQ)
- Метрики и мониторинг (Prometheus endpoint)
- Аутентификация и авторизация
- Кластеризация/репликация
- Web UI для мониторинга
- любые другие одобренные мной

## ТЕХНИЧЕСКИЕ ТРЕБОВАНИЯ

- **Язык программирования**: на выбор
- **Хранение данных**: файловая система, SQL, или любая NoSQL БД
- **Протокол взаимодействия**: на ваш выбор (Доп баллы за использование AMQP, STOMP или MQTT)
- **Документация**: Полная архитектурная схема, полная документация всех модулей системы
- **Деплой**: Docker контейнер с docker-compose (Доп баллы за сборки под любой linux/freebsd/unix ОС)

## КРИТЕРИИ ОЦЕНИВАНИЯ (10-балльная шкала)

### БАЗОВАЯ ФУНКЦИОНАЛЬНОСТЬ (максимум - 3 балла)
- **3 балла**: Полностью реализованы все базовые функции
- **2 балла**: Реализованы основные функции с небольшими недочетами
- **1 балл**: Реализована только часть базовой функциональности
- **0 баллов**: Базовые функции не работают или не реализованы

### ПЕРСИСТЕНТНОСТЬ И НАДЕЖНОСТЬ (максимум - 1 балл)
- **1 балл**: Сообщения сохраняются на диск, состояние восстанавливается после перезапуска, гарантия доставки реализована
- **0.5 балл**: Персистентность реализована частично (например, только сохранение без восстановления)
- **0 баллов**: Персистентность не реализована

### КЛИЕНТСКАЯ БИБЛИОТЕКА (максимум - 1 балл)

- **1 балл**: Полноценный SDK с удобным API, обработкой ошибок и переподключений
- **0.5 балл**: SDK реализован, но крайне примитивный
- **0 баллов**: Клиентская библиотека отсутствует или не работает


### ДОКУМЕНТАЦИЯ И СХЕМА (1 балл)
- **1 балл**: Подробная документация и схема (README, API docs)
- **0 баллов**: Документация или схема отсутствуют 

### ДОПОЛНИТЕЛЬНЫЕ ФУНКЦИИ
- **0.5-1 балл**: за каждый свой пункт, одобренный мной. Оценка пункта зависит от сложности его реализации

без доп функционала - максимум 6 баллов
система оценивания стандартная 10-ти бальная.

## ЭТАПЫ РАЗРАБОТКИ

### ЭТАП 1: ПРОЕКТИРОВАНИЕ И ДОКУМЕНТАЦИЯ

**Цель этапа**: Спроектировать архитектуру брокера сообщений и подготовить всю необходимую документацию перед началом разработки.

**Что должно быть готово:**
- Архитектурная схема системы (взаимодействие между модулями. если монолит - внутри монолита какие логические блоки/слои будут)
- Описание структуры данных (формат сообщений, метаданные, структура топиков/очередей)
- Технический стек (выбранные технологии, библиотеки, протокола взаимодействия - обоснование его выбора)
- План реализации (последовательность разработки модулей)
- README с описанием проекта, целями и требованиями


**Результат этапа**: Полный набор документации, позволяющий начать разработку без дополнительных вопросов по архитектуре.

---

### ЭТАП 2: БАЗОВАЯ ФУНКЦИОНАЛЬНОСТЬ

**Цель этапа**: Реализовать основную функциональность брокера сообщений.

**Что должно быть готово:**
- MVP продукта
- Реализация паттерна Publisher/Subscriber
- Поддержка топиков/очередей (создание, удаление, управление)
- Публикация сообщений в топики/очереди
- Подписка на топики/очереди
- Обработка множественных подписчиков
- Протокол взаимодействия внедрен и работает базово


**Результат этапа**: Работающий прототип брокера сообщений, способный обрабатывать сообщения в реальном времени.

---

### ЭТАП 3: РЕАЛИЗАЦИЯ ДОП ФУНКЦИЙ И БЕТА РЕЛИЗ

**Цель этапа**: Получить готовый продукт с частью доп функционала

**Что должно быть готово:**
- Сервис работает и максимально близок к релизу


**Результат этапа**: Надежный брокер с набором доп функций.

---

### ЭТАП 4: ДОПОЛНИТЕЛЬНЫЕ ФУНКЦИИ И ФИНАЛИЗАЦИЯ

**Цель этапа**: Реализовать дополнительные функции (по выбору) и сборка/деплой + тестовая среда.

**Что должно быть готово:**
- реализовать по желанию оставшиеся доп функции
- собрать продукт или обернуть в докер
- подготовить тестовую среду для демо (три микросервиса тестовых примитивных с подробным логом событий)
- проведение демо для сдачи финального продукта


**Результат этапа**: Финальная сдача продукта

**Примечания по этапам:**
- На первых двух этапах можете вносить изменения в архитектуру, если это обосновано и не полное ее переписывание (опять же все индивидуально)
- Дополнительные функции можно реализовывать на любом этапе, но основная оценка за них будет на этапе 4
- Важно показывать прогресс на каждом этапе, даже если что-то не полностью готово или не запускается
