# Технический стек

## Язык: Python 3.11+

**Обоснование:** Python — основной язык курса. Версия 3.11+ выбрана за:
- Зрелый `asyncio` с `TaskGroup` для управления корутинами
- `dataclasses` для описания моделей данных
- Типизация (`str | None` вместо `Optional[str]`) для читаемости
- Встроенный `sqlite3` — без внешних зависимостей для персистентности

## Сетевой уровень: asyncio TCP

**Выбор:** `asyncio.start_server()` с потоковым API (`StreamReader` / `StreamWriter`).

**Обоснование:**

| Вариант | Плюсы | Минусы | Решение |
|---------|-------|--------|---------|
| **asyncio TCP** | Минимум зависимостей, push-модель, низкие накладные расходы | — | **Выбран** |
| HTTP + WebSocket | Удобная отладка через curl/браузер | Два протокола, больше кода, overhead HTTP-заголовков | Отклонён |
| gRPC | Типизированные контракты | Тяжёлая зависимость, избыточно для учебного проекта | Отклонён |

**Почему не HTTP:** Pub/Sub требует push-доставки — сервер должен сам отправлять сообщения подписчику. В HTTP это возможно только через WebSocket или long polling, что добавляет сложность. TCP-соединение с `asyncio` естественно поддерживает двустороннюю коммуникацию.

## Протокол: STOMP 1.2

**Выбор:** STOMP (Simple Text Oriented Messaging Protocol), версия 1.2.

**Обоснование:**

| Вариант | Спецификация | Формат | Сложность реализации | Решение |
|---------|-------------|--------|---------------------|---------|
| **STOMP 1.2** | 15 страниц | Текстовый | Низкая | **Выбран** |
| Свой JSON-протокол | — | JSON | Низкая, но нет совместимости | Отклонён |
| AMQP 0-9-1 | 130 страниц | Бинарный | Очень высокая | Отклонён |
| MQTT 3.1.1 | 80 страниц | Бинарный | Средняя | Отклонён |

**Почему STOMP:**
- **Текстовый** — можно отлаживать через `telnet` и читать в логах глазами
- **Стандарт** — существуют готовые клиенты на всех языках (Python, Java, JS, Go)
- **Простой** — парсинг STOMP-фрейма ~50 строк кода без внешних библиотек
- **Достаточный** — покрывает все нужные операции: SEND, SUBSCRIBE, ACK, NACK
- **Совместимость** — можно подключиться любым STOMP-клиентом, а не только нашим SDK

**Почему не AMQP:** 130 страниц бинарного протокола, ~43 метода. Один handshake — это 6 бинарных фреймов. Нет ни одной простой реализации AMQP-сервера на Python.

**Почему не MQTT:** Бинарный протокол с переменной длиной заголовков. Реалистичен, но сложнее STOMP без дополнительной пользы для нашего проекта.

**Формат фрейма:**
```
COMMAND\n
header1:value1\n
header2:value2\n
\n
body\0
```

Парсинг: читаем байты до `\0`, разделяем на строки, первая строка — команда, далее заголовки до пустой строки, остаток — тело. Реализуется без внешних библиотек.

## Хранение данных: SQLite + WAL

**Выбор:** `sqlite3` (stdlib) + `aiosqlite` (асинхронная обёртка).

**Обоснование:**

| Вариант | Плюсы | Минусы | Решение |
|---------|-------|--------|---------|
| **SQLite** | Встроен в Python, ACID, SQL-запросы, один файл | Один writer одновременно | **Выбран** |
| PostgreSQL | Полноценный сервер, конкурентная запись | Внешняя зависимость, docker для запуска | Отклонён |
| Файловая система (append log) | Максимальная скорость записи | Сложность реализации compaction, индексов | Отклонён |
| Redis | Быстрый, pub/sub из коробки | Это уже готовый брокер, теряется учебный смысл | Отклонён |

**Режим WAL (Write-Ahead Log):**
- Позволяет читать данные во время записи (concurrent reads)
- Записи не блокируют чтение — важно для `check_ack_timeouts`, который сканирует БД, пока приходят новые сообщения
- Включается один раз: `PRAGMA journal_mode=WAL`

**Оптимизирующие PRAGMA:**
```sql
PRAGMA journal_mode = WAL;         
PRAGMA synchronous = NORMAL;        
PRAGMA cache_size = -64000;         
PRAGMA temp_store = MEMORY;         
```

`synchronous = NORMAL` (не `OFF`) — защита от потери данных при аварийном завершении процесса. При `OFF` данные теряются если упадёт ОС.

## Деплой: Docker

```dockerfile
FROM python:3.12-slim
WORKDIR /app
COPY requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt
COPY . .
EXPOSE 9090
VOLUME /app/data
CMD ["python", "-m", "pybroker.server"]
```

- **`python:3.12-slim`** — минимальный образ (~150 МБ vs ~900 МБ для полного)
- **`VOLUME /app/data`** — SQLite база данных на примонтированном томе для сохранения между перезапусками
- **`docker-compose.yml`** — брокер + три тестовых микросервиса (publisher, subscriber_a, subscriber_b)

## Почему не X?

| Технология | Причина отказа |
|-----------|----------------|
| FastAPI / Flask | HTTP-фреймворки не нужны для TCP-сервера |
| Celery | Это task queue поверх существующего брокера (RabbitMQ/Redis) |
| ZeroMQ | Бройкерлесс (peer-to-peer), не подходит по условию задачи |
| Pika (AMQP) | Клиент для RabbitMQ, а нам нужен свой сервер |
| Kafka-python | Клиент для Kafka, не сервер |
