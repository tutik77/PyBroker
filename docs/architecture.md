# Архитектура PyBroker

## Обзор

PyBroker — асинхронный брокер сообщений с протоколом STOMP 1.2. Архитектура разделена на четыре слоя, у каждого своя зона ответственности.

## Общая схема системы

```

│        │                │                │                       │
│        └────────────────┼────────────────┘                       │
│                    SDK (client.py)                                │

│                                                                  │
│  ┌─────────────────────────────────────────────────────────────┐ │
│  │  1. ТРАНСПОРТНЫЙ СЛОЙ (server.py, connection.py)            │ │
```

## Слои и их ответственность

### 1. Транспортный слой (`server.py`, `connection.py`)

**Задача:** управление TCP-соединениями.

- `server.py` — запуск `asyncio.start_server()`, приём новых подключений
- `connection.py` — обёртка над `StreamReader`/`StreamWriter`, хранит состояние клиента (ID, подписки, привязанные очереди)

Каждое подключение обрабатывается отдельной корутиной. Поскольку `asyncio` однопоточный, разделяемые структуры данных не требуют блокировок — корутины переключаются только на `await`.

```
Client A ──TCP──► [Connection A] ──► корутина handle_client(A)
Client B ──TCP──► [Connection B] ──► корутина handle_client(B)
Client C ──TCP──► [Connection C] ──► корутина handle_client(C)
                                           │
                                    asyncio event loop
                                    (один поток, без блокировок)
```

### 2. Протокольный слой (`protocol.py`)

**Задача:** парсинг и сборка STOMP-фреймов.

STOMP — текстовый протокол. Фрейм состоит из команды, заголовков и тела:
```
COMMAND\n
header1:value1\n
header2:value2\n
\n
body\0
```

- Парсинг: читаем из TCP-потока до нулевого байта `\0`, разбираем по строкам
- Первая строка — команда (`SEND`, `SUBSCRIBE`, `ACK`, ...)
- Далее заголовки в формате `ключ:значение`, до пустой строки
- После пустой строки — тело (опционально)
- Максимальный размер фрейма: 1 МБ (защита от переполнения)

### 3. Слой бизнес-логики (`broker.py`, `topic.py`, `queue.py`)

**Задача:** маршрутизация сообщений между издателями и потребителями.

```
                    broker.py
                  ┌───────────┐
                  │  Broker   │
                  │           │
          ┌───────┤  route()  ├────────┐
          │       │           │        │
          ▼       └───────────┘        ▼
    ┌──────────┐                ┌──────────┐
    │  Topics  │                │  Queues  │
    │          │                │          │
    │ /topic/* │                │ /queue/* │
    └──────────┘                └──────────┘
```

Маршрутизация по заголовку `destination` в STOMP-фрейме:
- `/topic/logs.error` → pub/sub (fan-out всем подписчикам)
- `/queue/tasks` → очередь (одному потребителю)

**Pub/Sub (topic.py):**
- Топик хранит множество (`set`) подписчиков (объектов `Connection`)
- При публикации — fan-out: сообщение отправляется **всем** подписчикам
- Поддержка wildcard-паттернов: `/topic/logs.*` матчит `/topic/logs.error` (один уровень)
- Подписчик, отключившийся, автоматически удаляется из множества

**Очереди (queue.py):**
- Очередь хранит список сообщений (`deque`) и список потребителей
- Поддержка FIFO (`popleft`) и LIFO (`pop`) — тип задаётся при создании
- При получении — round-robin: сообщение отправляется **одному** потребителю
- Двухфазная доставка: сообщение переходит в статус `IN_FLIGHT`, ждёт `ACK`
- Если `ACK` не получен за `visibility_timeout` — сообщение возвращается в очередь

```
Жизненный цикл сообщения в очереди:

                                ack()
  SEND ──► READY ──► IN_FLIGHT ──────► удалено
              ▲           │
              └───────────┘
           timeout / nack()
```

### 4. Слой хранения (`storage.py`)

**Задача:** персистентность сообщений очередей.

- SQLite в режиме WAL (Write-Ahead Log) для конкурентных чтений
- Асинхронный доступ через `aiosqlite`
- Сообщения очередей записываются на диск (Pub/Sub — только in-memory, т.к. подписчики получают сообщения в реальном времени)
- При старте брокера: загрузка `READY` и `IN_FLIGHT` сообщений из БД в память

Слой хранения изолирован за интерфейсом — при необходимости SQLite можно заменить на другой бэкенд без изменения бизнес-логики.

## Взаимодействие модулей (пример: публикация в топик)

```
Клиент                     Сервер
  │                          │
  │  SEND                    │
  │  destination:/topic/logs │
  │  content-type:text/plain │
  │                          │
  │  disk full\0             │
  │─────────────────────────►│
  │                          │
  │                    protocol.py:  парсит STOMP-фрейм → команда SEND
  │                    broker.py:    destination начинается с /topic/
  │                                 → route в topic_manager
  │                    topic.py:     fan-out → subscriber_a, subscriber_b
  │                          │
  │  RECEIPT                 │
  │  receipt-id:msg-1\0      │
  │◄─────────────────────────│
  │                          │
  │                    Каждому подписчику отправляется:
  │                    MESSAGE
  │                    destination:/topic/logs
  │                    message-id:uuid-xxx
  │
  │                    disk full\0
```

## Взаимодействие модулей (пример: consume из очереди)

```
Клиент                        Сервер
  │                             │
  │  SUBSCRIBE                  │
  │  destination:/queue/tasks   │
  │  id:sub-1                   │
  │  ack:client\0               │
  │────────────────────────────►│
  │                             │
  │                       broker.py: регистрирует consumer для очереди tasks
  │                             │
  │  ... (позже, когда в очереди появляется сообщение) ...
  │                             │
  │                       queue.py:    deque.popleft() → msg
  │                                    in_flight[msg.id] = (msg, now())
  │                       storage.py:  UPDATE status=IN_FLIGHT
  │                             │
  │  MESSAGE                    │
  │  destination:/queue/tasks   │
  │  message-id:uuid-xxx        │
  │  subscription:sub-1         │
  │                             │
  │  {"job": "resize"}\0        │
  │◄────────────────────────────│
  │                             │
  │  ACK                        │
  │  id:uuid-xxx\0              │
  │────────────────────────────►│
  │                             │
  │                       queue.py:    del in_flight[msg.id]
  │                       storage.py:  DELETE FROM messages
```
